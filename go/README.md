# Go memcache loader

Домашнее задание по переписыванию конкурентного memcache loader с Python на Go.

## Что делает программа

Программа читает gzip-файлы формата `*.tsv.gz`, в которых каждая строка содержит:

- тип устройства (`idfa`, `gaid`, `adid`, `dvid`)
- device id
- latitude
- longitude
- список app id через запятую

Для каждой строки программа:

1. парсит запись;
2. собирает protobuf `UserApps`;
3. формирует ключ вида `<dev_type>:<dev_id>`;
4. отправляет запись в соответствующий memcache по типу устройства.

После обработки файл переименовывается с префиксом `.` — так же, как в исходной Python-версии.

---

## Структура решения

### Основные идеи

Решение переписано на Go с использованием идиоматичных для языка примитивов конкурентности:

- **goroutine** — отдельные воркеры на запись;
- **channel** — очередь задач для каждого memcache-адреса;
- **WaitGroup** — ожидание завершения всех воркеров;
- **atomic counters** — подсчёт статистики без общей блокировки.

### Поток обработки

1. Главная goroutine открывает `.tsv.gz` файл.
2. Последовательно читает строки и парсит их в `AppsInstalled`.
3. В зависимости от `dev_type` отправляет job в канал нужного memcache.
4. Воркеры читают задачи из каналов и пишут данные батчами через `SetMulti`.
5. После завершения чтения каналы закрываются.
6. Главная goroutine ждёт завершения воркеров и считает итоговую статистику.

## Зависимости

Используется только protobuf runtime:

- `google.golang.org/protobuf/proto`

Memcache-клиент реализован прямо в проекте через обычный TCP text protocol (`set`), поэтому отдельная внешняя библиотека для memcache не нужна.

Установка зависимостей:

```bash
go mod tidy
```

## Запуск

### Dry-run

Режим без реальной записи в memcache, удобен для проверки парсинга и логики:

```bash
./memc_loader --dry --pattern="/data/appsinstalled/*.tsv.gz"
```

### Обычный запуск

```bash
./memc_loader \
  --pattern="/data/appsinstalled/*.tsv.gz" \
  --idfa="127.0.0.1:33013" \
  --gaid="127.0.0.1:33014" \
  --adid="127.0.0.1:33015" \
  --dvid="127.0.0.1:33016"
```

### Самопроверка protobuf

```bash
./memc_loader --test
```

## Поддерживаемые флаги

- `--test` — запускает встроенную самопроверку protobuf round-trip;
- `--log` — путь к лог-файлу;
- `--dry` — dry-run, без реальной записи в memcache;
- `--pattern` — glob-шаблон входных файлов;
- `--idfa` — адрес memcache для `idfa`;
- `--gaid` — адрес memcache для `gaid`;
- `--adid` — адрес memcache для `adid`;
- `--dvid` — адрес memcache для `dvid`;
- `--workers` — число воркеров на один memcache-адрес;
- `--batch` — размер батча для `SetMulti`;
- `--queue-size` — размер буфера канала;
- `--timeout` — timeout сетевого обращения к memcache;
- `--retry` — число повторных попыток записи батча;
- `--retry-backoff` — пауза между ретраями.

## Обработка ошибок

В программе учитываются следующие типы ошибок:

- битая строка входного файла;
- неизвестный `dev_type`;
- невалидные координаты;
- невалидные `app_id`;
- ошибка сериализации protobuf;
- ошибка записи в memcache.

Если `app_id` не является числом, он пропускается.
Если координаты не парсятся, используются значения `0, 0`.

После обработки считается доля ошибок:

- если `error_rate < 0.01`, загрузка считается успешной;
- иначе логируется ошибка высокой доли брака.

## Отличия от Python-версии

### Что улучшено

1. **Конкурентность реализована проще и чище**  
   В Go для этого естественно использовать goroutine и channels.

2. **Меньше накладных расходов**  
   В подобных I/O-bound задачах Go обычно показывает лучшую производительность и стабильнее масштабируется по числу воркеров.

3. **Явное управление жизненным циклом воркеров**  
   Закрытие каналов и `WaitGroup` дают предсказуемое завершение программы.

4. **Батчевая запись в memcache**  
   Используется `SetMulti`, что уменьшает число сетевых операций.

5. **Проще поддерживать статистику**  
   Вместо общей структуры с ручной синхронизацией используются атомарные счётчики.

### Что было проблемным в исходной Python-версии

В предоставленной версии Python-кода есть как минимум две ошибки:

- опечатка `isidigit()` вместо `isdigit()`;
- в статистике объявлены поля `processed/errors`, но используются `ok/err`.

Из-за этого исходная реализация либо работает некорректно, либо требует дополнительной правки перед честным сравнением.

## Сравнение Go и Python

### Python

Плюсы:

- код обычно пишется быстрее;
- меньше шаблонного кода;
- удобно для прототипирования.

Минусы:

- конкурентность тяжелее и менее естественна для CPU/IO-heavy пайплайнов;
- выше накладные расходы на потоки и синхронизацию;
- проще допустить ошибки в очередях, shared state и статистике.

### Go

Плюсы:

- лёгкие goroutine;
- удобная модель channel-based concurrency;
- проще масштабировать producer/consumer pipeline;
- хорошая производительность на сетевых и многопоточных задачах;
- бинарник удобно собирать и запускать.

Минусы:

- больше инфраструктурного кода, чем в Python;
- строже типизация;
- нужно аккуратнее проектировать структуру пайплайна.

В рамках домашнего задания конкурентный memcache loader был успешно переписан с Python на Go.

- сохраняет логику исходного загрузчика;

- проще масштабируется;
